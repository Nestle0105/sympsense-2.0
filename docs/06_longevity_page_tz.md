# ТЗ: Longevity Dashboard

## Контекст

Sympsense — персональная система хранения медицинских данных одного пользователя.
Цель пользователя: дожить до 100 лет в хорошем качестве.

Текущая система реактивная: документы попадают после визитов к врачу.
Longevity-дашборд — первый проактивный слой: отслеживает маркеры и базовые метрики независимо от болезней.

Концептуальный макет дашборда уже согласован (см. сессию с Claude).

---

## Структура проекта (для ориентации)

```
data/canonical/facts/          — structured facts (ndjson/json)
data/canonical/documents/      — document registry
data/derived/reports/          — generated reports
apps/cli/src/sympsense/        — FastAPI app
  api_server.py                — все API endpoints
  patient_briefing.py          — пример сложной аналитики
```

Сервер запускается через `sympsense serve-api`, доступен на `http://127.0.0.1:8000`.

## Что нужно построить

### 1. Новые типы canonical-данных

Три новых типа файлов в `data/canonical/longevity/`:

**`longevity_checkup_<date>.json`** — разовый расширенный чекап:
- дата
- список показателей: название, значение, единицы, референс, отклонение (выше/ниже/норма)
- источник (клиника или самостоятельно)

**`baseline_measurement_<type>.json`** — физические baseline-показатели:
- тип: `vo2max` | `grip_strength` | `balance` | `body_composition` | `bone_density` | `cognitive`
- дата измерения
- значение и единицы
- метод измерения (лаборатория, тест, устройство)
- примечание

**`protocol_<name>.json`** — что пользователь делает для оптимизации:
- название
- категория: `exercise` | `sleep` | `nutrition` | `supplementation` | `medical`
- статус: `active` | `paused` | `completed`
- дата начала / окончания
- описание

---

### 2. Новые API endpoints

Все в `api_server.py`, добавить перед `return app`.

**`GET /v1/longevity/overview`**
Главный эндпоинт дашборда. Возвращает полный payload для отрисовки страницы.
Логика сборки — в отдельном модуле `longevity_overview.py`.

**`GET /v1/longevity/horsemen`**
Статус четырёх зон риска (см. ниже). Вычисляется из canonical-данных.

**`GET /v1/longevity/baselines`**
Список baseline-измерений из `data/canonical/longevity/baseline_*.json`.

**`GET /v1/longevity/protocols`**
Список активных и завершённых протоколов.

**`GET /v1/longevity/gaps`**
Что отсутствует: маркеры без данных, просроченные проверки, незакрытые пробелы.

**`GET /v1/longevity/screening-calendar`**
Скрининги: что пройдено, что предстоит, что просрочено.

## Структура страницы (сверху вниз)

### Шапка
- Заголовок: «Longevity overview»
- Подзаголовок: сколько документов, временной диапазон, дата последнего обновления
- Справа: цель «100 лет»

### Горизонт жизни (progress bar)
Визуальная полоса 0 → 100 лет.
- Текущий возраст отмечен (пользователь вводит год рождения в `configs/app.yaml` или longevity config)
- Цель — 100 лет
- Под полосой: 4 числа — лет данных в системе, маркеров отслеживается, зон под контролем, действий требуют внимания

### Четыре всадника (2×2 grid)

Каждая карточка содержит:
- иконка + название зоны
- badge со статусом (см. логику ниже)
- список 5 метрик: название + значение + цветной индикатор
- кнопка «Что сдать» (опционально)

**Зона 1: Сердечно-сосудистые**
Метрики: ЛПНП, ApoB, общий холестерин, АД, кальциевый скоринг

**Зона 2: Метаболическое здоровье**
Метрики: HbA1c, инсулин натощак, глюкоза, креатинин / СКФ, мочевина

**Зона 3: Онкология**
Метрики: ПСА, колоноскопия (дата), дерматоскопия (дата), УЗИ органов (дата), онкомаркеры

**Зона 4: Нейродегенерация**
Метрики: когнитивный baseline (дата), качество сна (источник данных), гомоцистеин, головные боли (из анамнеза), головокружения (из анамнеза)

### Физические показатели
6 метрик в 2 колонки:
VO2max, сила хвата, баланс на одной ноге, состав тела (DEXA), плотность костей, опорно-двигательный статус

Опорно-двигательный статус берётся из `current_state_v1.json` (активные состояния по МКБ M).

### Воспаление и иммунитет
Метрики: CRP высокочувствительный, моноциты (с трендом), эозинофилы, гомоцистеин, ферритин

Данные по моноцитам и эозинофилам — из `lab_results_v1.ndjson`.

### Гормоны и восстановление
Метрики: тестостерон, ТТГ, витамин D, HRV / сон, репродуктивный статус

Репродуктивный статус (варикоцеле, тератозооспермия) — из `condition_mentions_v1.ndjson`.

### Приоритетные действия
Нумерованный список, сортировка по приоритету (red → amber → gray).
Каждый пункт: номер + заголовок + пояснение + badge с приоритетом.

### Что нужно добавить (gaps)
Теги-плашки со списком отсутствующих типов данных.
Берётся из `/v1/longevity/gaps`.

## Логика вычисления статусов

### Статус маркера (для каждой строки)
- `green` — есть данные, значение в норме, измерено менее 365 дней назад
- `amber` — есть данные, но: значение выше нормы ИЛИ измерено более 365 дней назад
- `red` — значение выше нормы И измерено более 365 дней назад (устаревшее отклонение)
- `gray` — данных нет вообще

### Статус зоны (badge на карточке всадника)
- `green` — все метрики green
- `amber` — хотя бы одна amber, нет red
- `red` — хотя бы одна red
- `нет данных` — все метрики gray

### Цвета badge
- `red` → «требует внимания»
- `amber` → «пробелы данных» или «устарело»
- `green` → «под контролем»
- `gray` → «нет данных»

## Откуда брать данные (маппинг)

**Из существующих canonical-данных:**

| Что нужно | Откуда брать |
|-----------|-------------|
| ЛПНП, холестерин, мочевина, креатинин, моноциты, эозинофилы | `lab_results_v1.ndjson` — поля `analyte_name`, `value_text`, `abnormal_flag`, `event_date` |
| Активные состояния (шея, плечо и др.) | `current_state_v1.json` — `active_conditions`, `long_term_conditions` |
| Головные боли, головокружения в анамнезе | `condition_mentions_v1.ndjson` — поиск по `condition_text` |
| Варикоцеле, тератозооспермия | `condition_mentions_v1.ndjson` |
| Даты последних документов по специальностям | `batch_01_registry_active.json` — поле `doc_type`, `event_date` |

**Из новых longevity-файлов** (будут пустые при первом запуске, страница это обрабатывает):
- baselines → `data/canonical/longevity/baseline_*.json`
- protocols → `data/canonical/longevity/protocol_*.json`
- checkups → `data/canonical/longevity/longevity_checkup_*.json`

## Скрининговый календарь

Рекомендуемые скрининги для мужчин (жёстко прописаны в логике, не из данных):

| Скрининг | С какого возраста | Периодичность |
|----------|------------------|---------------|
| Колоноскопия | 40–45 лет | каждые 10 лет |
| Кальциевый скоринг коронаров | 35–40 лет | однократно, затем по результату |
| ПСА (простата) | 40 лет | ежегодно |
| Дерматоскопия | 30 лет | ежегодно |
| УЗИ органов брюшной полости | 35 лет | каждые 3–5 лет |
| Офтальмолог | 25 лет | ежегодно (миопия в анамнезе) |
| Стоматолог | любой | каждые 6 месяцев |

Год рождения пользователя берётся из `configs/app.yaml` (добавить поле `birth_year`).
Если поля нет — скрининги показываются без статуса «просрочено / предстоит».

## Что НЕ делать

- Не делать клинические выводы или рекомендации по лечению
- Не вычислять «биологический возраст» — нет достаточных данных
- Не удалять существующие canonical-данные
- Не трогать `patient_briefing.py` и смежную логику
- Не требовать обязательного наличия longevity-файлов — страница работает и без них, показывая gaps
